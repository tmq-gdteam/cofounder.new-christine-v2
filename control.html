Approval model

Draft, recommend, automate, approve.

The product story uses a clear control vocabulary so visitors can tell what is being prepared and what is waiting on them.

Conceptual control layer
DraftCreate a candidate messageThe system prepares a version before it goes anywhere.Edit
RecommendSuggest the next decisionThe founder chooses whether the recommendation is useful.Compare

Never invisible

These actions need permission.

The site avoids broad security promises. Instead, it names the kinds of actions founders usually want held for review.

Spending

Budget, subscription, or payment actions should not happen without founder approval.

Publishing

Public copy, launch posts, and website changes should be reviewed before going live.

Emailing

Outbound messages should be drafted and approved before being sent.

Code changes

Changes that affect product behavior should be reviewed through an explicit workflow.

Commitments

Promises to customers, users, partners, or investors should stay with the founder.

Access

Sensitive information should be treated as requiring careful founder-controlled boundaries.
